# Support

<p>
  <img src="../.github/assets/readme/section-bars/reporting.svg" alt="REPORTING" width="100%">
</p>

| What you need | Where to go |
|---|---|
| install or build problem | open a private repo issue with `python --version`, exact install command, and full stderr |
| evidence dispute | open an issue or PR with exact counter-evidence file paths |
| benchmark or falsification question | start with `../proofs/ENGINEERING_BLOCKERS.md` and include the specific artifact you are disputing |
| security vulnerability | `architects@zer0pa.ai`; do not open a public issue |
| commercial licensing | `architects@zer0pa.ai` |
| release automation or PyPI publisher setup | follow `../proofs/runbooks/TRUSTED_PUBLISHING_OPERATOR_STEPS.md`; this remains operator-only |
| docs correction | open a PR with the exact file path and the evidence that justifies the change |

<p>
  <img src="../.github/assets/readme/section-bars/response-expectations.svg" alt="RESPONSE EXPECTATIONS" width="100%">
</p>

- Reproducible reports with exact commands and exact evidence paths get the
  fastest response.
- Target acknowledgement: 48 hours; initial triage: 7 days.
- Security issues follow `../SECURITY.md`.
- Owner-gated items may remain paused until the required outside action happens.

What will not get a fast answer here:

- requests to bless the repo as public-ready without new proof
- vague disputes that do not include an evidence path
- requests to reinterpret historical bundles as current authority
